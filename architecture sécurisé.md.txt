# Projet : Déploiement d'une infrastructure sécurisée sur Microsoft Azure

## Présentation

Dans le cadre de mon apprentissage du cloud et de la cybersécurité, j'ai réalisé ce projet afin de mieux comprendre le fonctionnement d'une infrastructure sécurisée sur Microsoft Azure.

L'objectif était de créer un environnement composé de plusieurs machines virtuelles Linux capables de communiquer entre elles de manière sécurisée. J'ai également voulu mettre en place un reverse proxy , une répartition de charge et différentes mesures de sécurité afin de me rapprocher d'une architecture utilisée en entreprise.

Ce projet m'a permis de découvrir plusieurs services Azure et de mettre en pratique les notions de réseau, d'administration Linux et de sécurisation d'une infrastructure.

---

# Architecture de l'infrastructure

L'infrastructure est composée de quatre machines virtuelles Ubuntu, chacune ayant un rôle précis.

 Machine virtuelle 	 Rôle                                                                

 vpn-server         Permet un accès sécurisé des administrateurs en SSH                 
 reverse-proxy22    Héberge Nginx qui sert de proxy inverse et de répartiteur de charge 
 web-client1        Premier serveur web                                                 
 web-client2        Deuxième serveur web                                                

---

# Architecture réseau

Pour mieux sécuriser l'infrastructure, j'ai choisi de séparer les différents services dans plusieurs réseaux virtuels Azure et sous-réseaux.

* VNET-HUB

  * VPN-SUBNET
  * REVERSEPROXY-SUBNET
  * ADMIN-SUBNET
  *PROXY-SUBNET

* VNET-CLIENT1

  * WEB-SUBNET1

* VNET-CLIENT2

  * WEB-SUBNET2

L'idée était d'éviter que toutes les machines soient sur le même réseau et de mieux contrôler les communications entre elles.

---

# Sécurisation de l'infrastructure

J'ai essayé d'appliquer plusieurs bonnes pratiques de sécurité que j'ai apprises au cours de ce projet :

* authentification SSH avec une paire de clés ;
* utilisation des Network Security Groups (NSG) d'Azure ;
* configuration du pare-feu UFW sur les serveurs ;
* installation de Fail2Ban pour limiter les attaques par force brute ;
* communication entre les serveurs via leurs adresses IP privées ;
* limitation des ports ouverts vers Internet ;
* protection des serveurs web derrière un proxy inverse.

Même si cette infrastructure reste simple, elle m'a permis de mieux comprendre l'importance de la segmentation réseau et du contrôle des accès.

---

# Proxy inverse

Le serveur **reverse-proxy22** utilise Nginx comme point d'entrée de l'infrastructure.

Toutes les requêtes HTTP arrivent d'abord sur ce serveur avant d'être redirigées vers les serveurs web.

```
Internet
    │
    ▼
Reverse Proxy (Nginx)
    │
    ▼
web-client1 / web-client2
```

Ainsi, les serveurs web ne sont jamais accessibles directement depuis Internet.

---

# Répartition de charge

J'ai également configuré Nginx pour répartir automatiquement les requêtes entre les deux serveurs web grâce au module **upstream**.

Cela permet de mieux répartir la charge et de continuer à servir les utilisateurs même si un serveur est plus sollicité que l'autre.

# Améliorations possibles

Si je devais continuer ce projet, j'aimerais ajouter plusieurs fonctionnalités :

* utiliser Azure Bastion pour éviter l'ouverture du port SSH sur Internet ;
* remplacer le proxy par Azure Application Gateway ;
* ajouter un Web Application Firewall (WAF) ;
* mettre en place Azure Monitor pour la supervision ;
* stocker les secrets avec Azure Key Vault ;
* automatiser le déploiement avec Terraform ;
* créer un pipeline CI/CD avec GitHub Actions.

Ces améliorations me permettraient de me rapprocher davantage d'une architecture utilisée dans un environnement professionnel.
